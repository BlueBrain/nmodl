1 ## The NMODL Framework
2 [![Build Status](https://travis-ci.org/BlueBrain/nmodl.svg?branch=master)](https://travis-ci.org/BlueBrain/nmodl) [![Build Status](https://dev.azure.com/pramodskumbhar/nmodl/_apis/build/status/BlueBrain.nmodl?branchName=master)](https://dev.azure.com/pramodskumbhar/nmodl/_build/latest?definitionId=2&branchName=master) [![codecov](https://codecov.io/gh/BlueBrain/nmodl/branch/master/graph/badge.svg?token=A3NU9VbNcB)](https://codecov.io/gh/BlueBrain/nmodl)
3 
4 The NMODL Framework is a code generation engine for **N**EURON **MOD**eling **L**anguage ([NMODL](https://www.neuron.yale.edu/neuron/static/py_doc/modelspec/programmatic/mechanisms/nmodl.html)). It is designed with modern compiler and code generation techniques to:
5 
6 * Provide **modular tools** for parsing, analysing and transforming NMODL
7 * Provide **easy to use**, high level Python API
8 * Generate **optimised code** for modern compute architectures including CPUs, GPUs
9 * **Flexibility** to implement new simulator backends
10 * Support for **full** NMODL specification
11 
12 ### About NMODL
13 
14 Simulators like [NEURON](https://www.neuron.yale.edu/neuron/) use NMODL as a domain specific language (DSL) to describe a wide range of membrane and intracellular submodels. Here is an example of exponential synapse specified in NMODL:
15 
16 ```python
17 NEURON {
18  POINT_PROCESS ExpSyn
19  RANGE tau, e, i
20  NONSPECIFIC_CURRENT i
21 }
22 UNITS {
23  (nA) = (nanoamp)
24  (mV) = (millivolt)
25  (uS) = (microsiemens)
26 }
27 PARAMETER {
28  tau = 0.1 (ms) <1e-9,1e9>
29  e = 0 (mV)
30 }
31 ASSIGNED {
32  v (mV)
33  i (nA)
34 }
35 STATE {
36  g (uS)
37 }
38 INITIAL {
39  g = 0
40 }
41 BREAKPOINT {
42  SOLVE state METHOD cnexp
43  i = g*(v - e)
44 }
45 DERIVATIVE state {
46  g' = -g/tau
47 }
48 NET_RECEIVE(weight (uS)) {
49  g = g + weight
50 }
51 ```

106 ### Using the Python API
107 
108 Once the NMODL Framework is installed, you can use the Python parsing API to load NMOD file as:
109 
110 ```python
111 from nmodl import dsl
112 
113 examples = dsl.list_examples()
114 nmodl_string = dsl.load_example(examples[-1])
115 driver = dsl.NmodlDriver()
116 modast = driver.parse_string(nmodl_string)
117 ```
118 
119 The `parse_file` API returns Abstract Syntax Tree ([AST](https://en.wikipedia.org/wiki/Abstract_syntax_tree)) representation of input NMODL file. One can look at the AST by converting to JSON form as:
120 
121 ```python
122 >>> print (dsl.to_json(modast))
123 {
124  "Program": [
125  {
126  "NeuronBlock": [
127  {
128  "StatementBlock": [
129  {
130  "Suffix": [
131  {
132  "Name": [
133  {
134  "String": [
135  {
136  "name": "POINT_PROCESS"
137  }
138  ...
139 ```
140 Every key in the JSON form represent a node in the AST. You can also use visualization API to look at the details of AST as:
141 
142 ```
143 from nmodl import ast
144 ast.view(modast)
145 ```
146 
147 which will open AST view in web browser:
148 
149 ![ast_viz](https://user-images.githubusercontent.com/666852/57329449-12c9a400-7114-11e9-8da5-0042590044ec.gif "AST representation of expsyn.mod")
150 
151 The central *Program* node represents the whole MOD file and each of it's children represent the block in the input NMODL file. Note that this requires X-forwarding if you are using Docker image.
152 
153 Once the AST is created, one can use exisiting visitors to perform various analysis/optimisations. One can also easily write his own custom visitor using Python Visitor API. See [Python API tutorial](docs/notebooks/nmodl-python-tutorial.ipynb) for details.
154 
155 NMODL Frameowrk also allows to transform AST representation back to NMODL form as:
156 
157 ```python
158 >>> print (dsl.to_nmodl(modast))
159 NEURON {
160  POINT_PROCESS ExpSyn
161  RANGE tau, e, i
162  NONSPECIFIC_CURRENT i
163 }
164 
165 UNITS {
166  (nA) = (nanoamp)
167  (mV) = (millivolt)
168  (uS) = (microsiemens)
169 }
170 
171 PARAMETER {
172  tau = 0.1 (ms) <1e-09,1000000000>
173  e = 0 (mV)
174 }
175 ...
176 ```

178 ### High Level Analysis and Code Generation
179 
180 The NMODL Framework provides rich model introspection and analysis capabilities using [various visitors](https://bluebrain.github.io/nmodl/html/doxygen/group__visitor__classes.html). Here is an example of theoretical performance characterisation of channels and synapses from rat neocortical column microcircuit [published in 2015](https://www.cell.com/abstract/S0092-8674%2815%2901191-5):
181 
182 ![nmodl-perf-stats](https://user-images.githubusercontent.com/666852/57336711-2cc0b200-7127-11e9-8053-8f662e2ec191.png "Example of performance characterisation")
183 
184 To understand how you can write your own introspection and analysis tool, see [this tutorial](docs/notebooks/nmodl-python-tutorial.ipynb).
185 
186 Once analysis and optimization passes are performed, the NMODL Framework can generate optimised code for modern compute architectures including CPUs (Intel, AMD, ARM) and GPUs (NVIDIA, AMD) platforms. For example, C++, OpenACC, OpenMP, CUDA and ISPC backends are implemented and one can choose these backends on command line as:
187 
188 ```
189 $ nmodl expsyn.mod host --ispc acc --cuda sympy --analytic
190 ```
191 
192 Here is an example of generated [ISPC](https://ispc.github.io/) kernel for DERIVATIVE block :
193 
194 ```c++
195 export void nrn_state_ExpSyn(uniform ExpSyn_Instance* uniform inst, uniform NrnThread* uniform nt ...) {
196  uniform int nodecount = ml->nodecount;
197  const int* uniform node_index = ml->nodeindices;
198  const double* uniform voltage = nt->actual_v;
199 
200  int uniform start = 0;
201  int uniform end = nodecount;
202 
203  foreach (id = start ... end) {
204  int node_id = node_index[id];
205  double v = voltage[node_id];
206  inst->g[id] = inst->g[id] * vexp( -nt->dt / inst->tau[id]);
207  }
208 }
209 ```
210 
211 To know more about code generation backends, [see here](https://bluebrain.github.io/nmodl/html/doxygen/group__codegen__backends.html). NMODL Framework provides number of options (for code generation, optimization passes and ODE solver) which can be listed as:
212 
213 ```
214 $ nmodl -H
215 NMODL : Source-to-Source Code Generation Framework [version]
216 Usage: /path/<>/nmodl [OPTIONS] file... [SUBCOMMAND]
217 
218 Positionals:
219  file TEXT:FILE ... REQUIRED One or more MOD files to process
220 
221 Options:
222  -h,--help Print this help message and exit
223  -H,--help-all Print this help message including all sub-commands
224  --verbose=info Verbose logger output (trace, debug, info, warning, error, critical, off)
225  -o,--output TEXT=. Directory for backend code output
226  --scratch TEXT=tmp Directory for intermediate code output
227  --units TEXT=/path/<>/nrnunits.lib
228  Directory of units lib file
229 
230 Subcommands:
231 host
232  HOST/CPU code backends
233  Options:
234  --c C/C++ backend (true)
235  --omp C/C++ backend with OpenMP (false)
236  --ispc C/C++ backend with ISPC (false)
237 
238 acc
239  Accelerator code backends
240  Options:
241  --oacc C/C++ backend with OpenACC (false)
242  --cuda C/C++ backend with CUDA (false)
243 
244 sympy
245  SymPy based analysis and optimizations
246  Options:
247  --analytic Solve ODEs using SymPy analytic integration (false)
248  --pade Pade approximation in SymPy analytic integration (false)
249  --cse CSE (Common Subexpression Elimination) in SymPy analytic integration (false)
250  --conductance Add CONDUCTANCE keyword in BREAKPOINT (false)
251 
252 passes
253  Analyse/Optimization passes
254  Options:
255  --inline Perform inlining at NMODL level (false)
256  --unroll Perform loop unroll at NMODL level (false)
257  --const-folding Perform constant folding at NMODL level (false)
258  --localize Convert RANGE variables to LOCAL (false)
259  --global-to-range Convert GLOBAL variables to RANGE (false)
260  --localize-verbatim Convert RANGE variables to LOCAL even if verbatim block exist (false)
261  --local-rename Rename LOCAL variable if variable of same name exist in global scope (false)
262  --verbatim-inline Inline even if verbatim block exist (false)
263  --verbatim-rename Rename variables in verbatim block (true)
264  --json-ast Write AST to JSON file (false)
265  --nmodl-ast Write AST to NMODL file (false)
266  --json-perf Write performance statistics to JSON file (false)
267  --show-symtab Write symbol table to stdout (false)
268 
269 codegen
270  Code generation options
271  Options:
272  --layout TEXT:{aos,soa}=soa Memory layout for code generation
273  --datatype TEXT:{float,double}=soa Data type for floating point variables
274  --force Force code generation even if there is any incompatibility
275  --only-check-compatibility Check compatibility and return without generating code
276  --opt-ionvar-copy Optimize copies of ion variables (false)
277 ```

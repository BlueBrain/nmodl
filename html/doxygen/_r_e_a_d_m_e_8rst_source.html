1 The NMODL Framework
2 ===================
3 
4 |github workflow| |Build Status| |codecov| |CII Best Practices|
5 
6 The NMODL Framework is a code generation engine for **N**\ EURON
7 **MOD**\ eling **L**\ anguage
8 (`NMODL <https://www.neuron.yale.edu/neuron/static/py_doc/modelspec/programmatic/mechanisms/nmodl.html>`__).
9 It is designed with modern compiler and code generation techniques to:
10 
11 - Provide **modular tools** for parsing, analysing and transforming
12  NMODL
13 - Provide **easy to use**, high level Python API
14 - Generate **optimised code** for modern compute architectures
15  including CPUs, GPUs
16 - **Flexibility** to implement new simulator backends
17 - Support for **full** NMODL specification
18 
19 About NMODL
20 -----------
21 
22 Simulators like `NEURON <https://www.neuron.yale.edu/neuron/>`__ use
23 NMODL as a domain specific language (DSL) to describe a wide range of
24 membrane and intracellular submodels. Here is an example of exponential
25 synapse specified in NMODL:
26 
27 .. code:: python
28 
29  NEURON {
30  POINT_PROCESS ExpSyn
31  RANGE tau, e, i
32  NONSPECIFIC_CURRENT i
33  }
34  UNITS {
35  (nA) = (nanoamp)
36  (mV) = (millivolt)
37  (uS) = (microsiemens)
38  }
39  PARAMETER {
40  tau = 0.1 (ms) <1e-9,1e9>
41  e = 0 (mV)
42  }
43  ASSIGNED {
44  v (mV)
45  i (nA)
46  }
47  STATE {
48  g (uS)
49  }
50  INITIAL {
51  g = 0
52  }
53  BREAKPOINT {
54  SOLVE state METHOD cnexp
55  i = g*(v - e)
56  }
57  DERIVATIVE state {
58  g' = -g/tau
59  }
60  NET_RECEIVE(weight (uS)) {
61  g = g + weight
62  }

71 Try NMODL with Docker
72 ---------------------
73 
74 To quickly test the NMODL Framework’s analysis capabilities we provide a
75 `docker <https://www.docker.com>`__ image, which includes the NMODL
76 Framework python library and a fully functional Jupyter notebook
77 environment. After installing
78 `docker <https://docs.docker.com/compose/install/>`__ and
79 `docker-compose <https://docs.docker.com/compose/install/>`__ you can
80 pull and run the NMODL image from your terminal.
81 
82 To try Python interface directly from CLI, you can run docker image as:
83 
84 ::
85 
86  docker run -it --entrypoint=/bin/sh bluebrain/nmodl
87 
88 And try NMODL Python API discussed later in this README as:
89 
90 ::
91

130 Using the Python API
131 --------------------
132 
133 Once the NMODL Framework is installed, you can use the Python parsing
134 API to load NMOD file as:
135 
136 .. code:: python
137 
138  from nmodl import dsl
139 
140  examples = dsl.list_examples()
141  nmodl_string = dsl.load_example(examples[-1])
142  driver = dsl.NmodlDriver()
143  modast = driver.parse_string(nmodl_string)
144 
145 The ``parse_file`` API returns Abstract Syntax Tree
146 (`AST <https://en.wikipedia.org/wiki/Abstract_syntax_tree>`__)
147 representation of input NMODL file. One can look at the AST by
148 converting to JSON form as:
149 
150 .. code:: python
151 
152  >>> print (dsl.to_json(modast))
153  {
154  "Program": [
155  {
156  "NeuronBlock": [
157  {
158  "StatementBlock": [
159  {
160  "Suffix": [
161  {
162  "Name": [
163  {
164  "String": [
165  {
166  "name": "POINT_PROCESS"
167  }
168  ...
169 
170 Every key in the JSON form represent a node in the AST. You can also use
171 visualization API to look at the details of AST as:
172 
173 ::
174 
175  from nmodl import ast
176  ast.view(modast)
177 
178 which will open AST view in web browser:
179 
180 .. figure::
181  https://user-images.githubusercontent.com/666852/57329449-12c9a400-7114-11e9-8da5-0042590044ec.gif
182  :alt: ast_viz
183 
184  ast_viz
185 
186 The central *Program* node represents the whole MOD file and each of
187 it’s children represent the block in the input NMODL file. Note that
188 this requires X-forwarding if you are using Docker image.
189 
190 Once the AST is created, one can use exisiting visitors to perform
191 various analysis/optimisations. One can also easily write his own custom
192 visitor using Python Visitor API. See `Python API
193 tutorial <docs/notebooks/nmodl-python-tutorial.ipynb>`__ for details.
194 
195 NMODL Frameowrk also allows to transform AST representation back to
196 NMODL form as:
197 
198 .. code:: python
199 
200  >>> print (dsl.to_nmodl(modast))
201  NEURON {
202  POINT_PROCESS ExpSyn
203  RANGE tau, e, i
204  NONSPECIFIC_CURRENT i
205  }
206 
207  UNITS {
208  (nA) = (nanoamp)
209  (mV) = (millivolt)
210  (uS) = (microsiemens)
211  }
212 
213  PARAMETER {
214  tau = 0.1 (ms) <1e-09,1000000000>
215  e = 0 (mV)
216  }
217  ...

219 High Level Analysis and Code Generation
220 ---------------------------------------
221 
222 The NMODL Framework provides rich model introspection and analysis
223 capabilities using `various
224 visitors <https://bluebrain.github.io/nmodl/html/doxygen/group__visitor__classes.html>`__.
225 Here is an example of theoretical performance characterisation of
226 channels and synapses from rat neocortical column microcircuit
227 `published in
228 2015 <https://www.cell.com/abstract/S0092-8674%2815%2901191-5>`__:
229 
230 .. figure::
231  https://user-images.githubusercontent.com/666852/57336711-2cc0b200-7127-11e9-8053-8f662e2ec191.png
232  :alt: nmodl-perf-stats
233 
234  nmodl-perf-stats
235 
236 To understand how you can write your own introspection and analysis
237 tool, see `this
238 tutorial <docs/notebooks/nmodl-python-tutorial.ipynb>`__.
239 
240 Once analysis and optimization passes are performed, the NMODL Framework
241 can generate optimised code for modern compute architectures including
242 CPUs (Intel, AMD, ARM) and GPUs (NVIDIA, AMD) platforms. For example,
243 C++, OpenACC and OpenMP backends are implemented and one can choose
244 these backends on command line as:
245 
246 ::
247 
248  $ nmodl expsyn.mod sympy --analytic
249 
250 To know more about code generation backends, `see
251 here <https://bluebrain.github.io/nmodl/html/doxygen/group__codegen__backends.html>`__.
252 NMODL Framework provides number of options (for code generation,
253 optimization passes and ODE solver) which can be listed as:
254 
255 ::
256 
257  $ nmodl -H
258  NMODL : Source-to-Source Code Generation Framework [version]
259  Usage: /path/<>/nmodl [OPTIONS] file... [SUBCOMMAND]
260 
261  Positionals:
262  file TEXT:FILE ... REQUIRED One or more MOD files to process
263 
264  Options:
265  -h,--help Print this help message and exit
266  -H,--help-all Print this help message including all sub-commands
267  --verbose=info Verbose logger output (trace, debug, info, warning, error, critical, off)
268  -o,--output TEXT=. Directory for backend code output
269  --scratch TEXT=tmp Directory for intermediate code output
270  --units TEXT=/path/<>/nrnunits.lib
271  Directory of units lib file
272 
273  Subcommands:
274  host
275  HOST/CPU code backends
276  Options:
277  --c C/C++ backend (true)
278 
279  acc
280  Accelerator code backends
281  Options:
282  --oacc C/C++ backend with OpenACC (false)
283 
284  sympy
285  SymPy based analysis and optimizations
286  Options:
287  --analytic Solve ODEs using SymPy analytic integration (false)
288  --pade Pade approximation in SymPy analytic integration (false)
289  --cse CSE (Common Subexpression Elimination) in SymPy analytic integration (false)
290  --conductance Add CONDUCTANCE keyword in BREAKPOINT (false)
291 
292  passes
293  Analyse/Optimization passes
294  Options:
295  --inline Perform inlining at NMODL level (false)
296  --unroll Perform loop unroll at NMODL level (false)
297  --const-folding Perform constant folding at NMODL level (false)
298  --localize Convert RANGE variables to LOCAL (false)
299  --global-to-range Convert GLOBAL variables to RANGE (false)
300  --localize-verbatim Convert RANGE variables to LOCAL even if verbatim block exist (false)
301  --local-rename Rename LOCAL variable if variable of same name exist in global scope (false)
302  --verbatim-inline Inline even if verbatim block exist (false)
303  --verbatim-rename Rename variables in verbatim block (true)
304  --json-ast Write AST to JSON file (false)
305  --nmodl-ast Write AST to NMODL file (false)
306  --json-perf Write performance statistics to JSON file (false)
307  --show-symtab Write symbol table to stdout (false)
308 
309  codegen
310  Code generation options
311  Options:
312  --layout TEXT:{aos,soa}=soa Memory layout for code generation
313  --datatype TEXT:{float,double}=soa Data type for floating point variables
314  --force Force code generation even if there is any incompatibility
315  --only-check-compatibility Check compatibility and return without generating code
316  --opt-ionvar-copy Optimize copies of ion variables (false)
